## 1. 프로젝트 개요

EduPulse는 코딩, 보안, 게임, 아트 분야 학원의 **수강 신청 수요를 사전 예측**하는 AI 솔루션이다. 수요 예측 결과를 운영(강사 배정), 마케팅(광고 타이밍), 전략(신규 강좌 시뮬레이션) 세 방향으로 연결하여 학원 운영자의 의사결정을 지원한다.

| 항목 | 내용 |

| 공모전 주제 | AI 활용 차세대 교육 솔루션 |
| 개발 기간 | 약 2주 (2026-04-07 ~ 2026-04-10) |
| 개발 인원 | 2인 |

### 2.2 단계별 핵심 의사결정

**아이디어 기획 (Day 1):** AI가 14개 이상의 솔루션 아이디어를 발산적으로 생성하고, 비즈니스 임팩트와 구현 가능성을 기준으로 "수강 신청 수요 예측"을 선정하였다. 단일 예측 모델을 운영, 마케팅, 전략 세 축으로 연결하는 서비스 구조를 확정하였다.

**계획 수립 (Day 1-2):** Ralplan 합의 루프에서 Architect가 Droplet 2GB에서 3모델 동시 서빙 시 메모리 초과 문제를 사전 발견하고, Critic이 pydantic-settings 누락 등 실행 블로커를 탐지하였다. 초기 6-Phase 계획이 4-Phase MVP로 압축되었다.

**백엔드 구현 (Day 2-4):** Team 워크플로우로 복수 에이전트가 TaskList를 공유하며 API, DB, 합성 데이터 생성기를 병렬 구현하였다. Contract-First 접근으로 Pydantic 스키마를 먼저 정의하고 구현을 진행하였다.

**모델 개선 (Day 3-5):** 기수 단위(96행)에서 주간 단위(1668행)로 데이터 해상도를 전환하고, 분야별 분리 학습, feature-aware 데이터 증강, Early Stopping을 적용하여 3개 모델 모두 MAPE를 개선하였다.

**프론트엔드 (Day 4-7):** React 18 + Vite 기반으로 Dashboard, Simulator, Marketing, Operations, Market 5개 페이지를 구현하고, Adapter 패턴(Mock/Hybrid/Real)으로 백엔드 연동 전환을 지원하였다. 18+3건의 UX 개선과 Docker multi-stage 빌드를 완료하였다.

**서버 배포 (Day 7-8):** Code-Reviewer가 20건의 배포 이슈를 발견하고, 그중 6건의 CRITICAL(경로 하드코딩, CORS, 크레덴셜 노출 등)을 포함하여 전수 수정 후 Portainer를 통해 배포하였다.

## 3. AI 도구 활용 체계

### 3.1 도구 진화 과정

프로젝트 진행에 따라 AI 활용 도구가 3단계로 진화하였다.

**1단계 — Claude (claude.ai):** 아이디어 발굴, 서비스 범위 정의, 데이터 설계에 대화형으로 활용하였다. 사용자가 공모전 주제와 제약 조건을 제공하면 AI가 선택지를 제시하고, 반복 대화를 통해 의사결정을 구체화하는 방식이다.

**2단계 — Claude Code:** CLI 환경에서 파일 읽기, 코드 생성, 편집, 테스트 실행을 수행하였다. 단일 에이전트가 계획부터 구현, 검증까지 모두 담당하는 구조이다.

**3단계 — oh-my-claudecode (OMC):** 멀티에이전트 오케스트레이션 레이어를 통해 계획, 구현, 검증을 역할별로 분리하였다. 자기 승인 방지, 모델 라우팅, 컨텍스트 격리 등 품질 강제 메커니즘을 적용하였다.

### 3.2 사용한 OMC 워크플로우

| 워크플로우 | 설명 | 적용 단계 |
|----------|------|----------|
| Ralplan | Planner, Architect, Critic 3자 합의 루프 | 계획 수립, 모델 개선 |
| Team | 복수 에이전트가 TaskList 공유 병렬 실행 | 백엔드, 프론트엔드 |
| Ralph | 자기참조 실행(Executor)과 검증(Verifier) 루프 | 한글화, lint 수정 |
| Autopilot | 계획, 실행, 검증을 완전 자율로 수행 | 단일 기능 구현 |

**Ralplan 데이터 흐름:**

```
사용자 제약 조건
    |
    v
[Planner] ── 초안 작성 (RALPLAN-DR + Phase + ADR)
    |
    v
[Architect] ── 아키텍처 리뷰 (Steelman 반론, Tradeoff 분석)
    |
    v                          ※ 순차 실행 필수
[Critic] ── 품질 평가 (5기준 Pass/Fail)
    |
    +-- APPROVE --> 실행 착수
    |
    +-- ITERATE --> Planner 수정 --> Architect --> Critic (최대 5회)
```

**Team 데이터 흐름:**

```
[Team Lead] ── TaskList 관리
    |
    +-- [Agent A] ── Task 1 (Backend API)
    +-- [Agent B] ── Task 2 (Frontend Page)
    +-- [Agent C] ── Task 3 (Tests)
```

### 3.3 에이전트 역할 분리

| 에이전트 | 역할 | 모델 | 핵심 기여 |
|---------|------|------|----------|
| Planner | 계획 수립 | Opus | 4-Phase MVP 구조, 일별 태스크 분해 |
| Architect | 설계 검증 | Opus (READ-ONLY) | Droplet 메모리 초과 사전 발견, Steelman 반론 |
| Critic | 품질 평가 | Opus (READ-ONLY) | pydantic-settings 누락 등 블로커 탐지 |
| Executor | 코드 구현 | Sonnet / Opus | 실제 파일 생성, 수정 |
| Code-Reviewer | 코드 리뷰 | Sonnet | 보안 취약점, 로직 결함 탐지 |
| Verifier | 완료 검증 | Sonnet | scope creep 탐지, 테스트 통과 확인 |

## 4. 핵심 설계 원칙 — AI 자기 승인 방지

본 프로젝트에서 AI 협업의 품질을 강제한 핵심 메커니즘은 **작성과 검증의 분리**이다.

**원칙 1 — 구현과 리뷰 분리:** Executor가 작성한 코드를 Code-Reviewer가 독립적으로 리뷰한다. 동일 에이전트가 작성과 리뷰를 겸하면 확인 편향(confirmation bias)이 발생하므로, 이를 구조적으로 차단한다.

**원칙 2 — 계획과 평가 분리:** Planner가 수립한 계획을 Architect와 Critic이 독립적으로 평가한다. Planner는 자기 계획을 승인할 수 없다.

**원칙 3 — READ-ONLY 에이전트:** Architect와 Critic은 코드를 수정할 수 없다. 읽기만 가능하므로 "문제를 발견하고 직접 고치면서 리뷰를 통과시키는" 행위를 원천 차단한다.

**원칙 4 — 동일 컨텍스트 자기 승인 불가:** 하나의 에이전트가 작성 후 같은 컨텍스트에서 "완료"를 선언할 수 없다. 반드시 Verifier를 거쳐야 한다.

이 원칙들이 실제로 작동한 사례:
- Architect가 "합성 데이터 3모델 비교는 자기순환적 허구"라는 Steelman 반론을 제시하여, 3모델 전체 구현에서 XGBoost 집중 전략으로 전환
- Verifier가 프론트엔드 구현 시 scope creep(React 버전 임의 변경, 백엔드 코드 오염)을 탐지하여 차단
- Code-Reviewer가 배포 직전 CORS 설정 누락, 크레덴셜 하드코딩 등 6건의 CRITICAL 이슈를 발견

## 5. 정량적 성과

### 5.1 코드 규모

| 영역 | 파일 수 | 주요 내용 |
|------|--------|----------|
| 백엔드 (edupulse/) | ~35 | FastAPI, 모델 3종, 전처리, 합성 데이터 생성기 |
| 프론트엔드 (frontend/) | ~25 | React 5페이지, 컴포넌트, Adapter 패턴 |
| 테스트 (tests/) | ~8 | pytest 30+ 테스트 케이스 |
| 인프라 | ~5 | Dockerfile, docker-compose, nginx |
| 문서 | ~10 | CLAUDE.md, README, AI 리포트 7건 |

### 5.2 모델 성능

| 모델 | 초기 MAPE | 최종 MAPE | 개선량 |
|------|----------|----------|--------|
| XGBoost | 18.55% | 16.20% | -2.35%p |
| Prophet | 52.93% | 45.44% | -7.49%p |
| LSTM | 46.72% | 33.81% | -12.91%p |

**Phase별 개선 경로:**

| 모델 | 원본 (96행) | Phase 1 (1044행) | Phase 2 (구조 개선) | Phase 3 (최종) |
|------|----------:|------------------:|------------------:|--------------:|
| XGBoost | 18.55% | 24.64% (악화) | 17.46% | 16.20% |
| Prophet | 52.93% | 50.19% | 45.44% | 45.44% |
| LSTM | 46.72% | 32.82% | 24.30% | 33.81% |

XGBoost는 주간 전환 시 MAPE가 일시 악화(24.64%)되었으나, field_encoded 피처 추가와 노이즈 축소로 16.20%까지 회복하였다. Phase 1에서의 악화는 모델 성능 저하가 아니라 MAPE 지표의 소수값 편향(동일 1명 오차가 기수 단위 5% vs 주간 단위 25%)에 기인한다.

Prophet은 분야별 분리 학습이 핵심 개선 요인이었다. 4개 분야가 혼합된 시계열을 단일 모델로 학습하면 분야 전환 지점에서 거짓 패턴을 학습하는 문제가 있었고, 분야별 개별 모델로 전환하여 7.49%p를 개선하였다.

LSTM은 데이터 확대(96행에서 1668행)와 feature-aware 증강이 가장 큰 기여를 하였다. Phase 3의 33.81%는 8년(1668행) 데이터 기준이며, COVID 급감/회복 등 복잡한 트렌드가 추가된 조건이다. 실제 `train()`에서는 augmentation + Early Stopping + LR Scheduler가 모두 적용되므로 실전 성능은 evaluate 수치보다 우수할 것으로 판단된다.

## 6. AI 협업의 효과와 한계

### 6.1 효과

**리스크 조기 발견:** 구현 전 계획 단계에서 아키텍처 이슈를 발견하였다. 대표적으로 Architect가 Droplet 2GB에서 PostgreSQL + FastAPI + 3모델 동시 서빙 시 메모리 초과가 발생함을 계산하여, 구현 후 아키텍처 재설계라는 비용을 사전에 회피하였다.

**품질 강제:** 자기 승인 방지 구조로 확인 편향을 최소화하였다. Critic이 "대안이 2개뿐"이라는 이유로 FAIL 판정을 내려 MVP-First(Option C) 검토를 강제한 결과, 최적 전략이 발견되었다.

**병렬 개발:** Team 워크플로우로 백엔드와 프론트엔드를 동시에 진행하였다. 복수 에이전트가 TaskList를 공유하면서 독립적인 파일을 병렬 수정하여 구현 시간을 단축하였다.

**반복 자동화:** Ralph 루프로 한글화, lint 수정 등 반복 작업을 자동 완료하였다. Executor가 수정하고 Verifier가 검증하는 루프가 모든 항목이 통과할 때까지 자동으로 순환하였다.

**문서화 동반:** 계획서, ADR(Architecture Decision Record), 핸드오프 문서가 개발과 동시에 생성되었다. 별도의 문서화 시간을 할애하지 않고도 의사결정 근거가 보존되었다.

### 6.2 한계

**합성 데이터 자기순환:** 생성기가 심어놓은 계절성과 트렌드를 모델이 재발견하는 구조이다. MAPE 수치는 모델 성능이 아니라 합성 데이터 설계 품질을 반영할 뿐이며, 실제 학원 데이터 없이는 모델의 실용적 가치를 검증할 수 없다.

**인프라 조작 불가:** Portainer, Nginx Proxy Manager 등 Web UI 기반 설정은 AI가 직접 수행할 수 없다. 배포 가이드 문서를 생성하여 사용자가 수동으로 설정하는 방식으로 우회하였다.

**일정 추정 정확도:** AI가 제시한 일정(10일 + 2일 버퍼)은 실제 진행과 차이가 발생할 수 있다. "Day 5에 XGBoost 학습 완료"와 같은 구체적 일정은 예측이지 약속이 아니다.

**공모전 심사 기준 불확실:** "1개 모델을 깊이 있게" 구현하는 전략과 "3개 모델을 폭넓게" 비교하는 전략 중 심사위원이 어느 쪽을 높이 평가하는지는 AI가 판단할 수 없는 영역이다.

## 9. 결론

2주, 2인으로 풀스택 AI 예측 솔루션을 완성할 수 있었던 핵심 요인은 **AI와의 체계적 협업**이다.

단순한 코드 생성을 넘어, 기획에서 설계, 구현, 검증, 배포에 이르는 전 과정에서 AI가 공동 의사결정자로 참여하였다. 아이디어 단계에서는 발산적 사고를 지원하고, 계획 단계에서는 합의 루프로 완성도를 높이며, 구현 단계에서는 병렬 실행으로 속도를 확보하고, 배포 단계에서는 독립적 리뷰로 품질을 강제하였다.

특히 두 가지 구조적 장치가 차별화 요소이다. 첫째, **멀티에이전트 분리(작성/검증)**로 확인 편향을 방지하고, 자기 승인 없이 독립적 검증을 강제하였다. 둘째, **합의 루프(Ralplan)**로 계획의 완성도를 2회 반복만에 5개 품질 기준 전부 통과 수준으로 끌어올렸다.

본 프로젝트는 AI를 "코드를 대신 작성해주는 도구"가 아닌, **기획부터 배포까지 전 과정의 공동 의사결정자**로 활용한 사례이다. 멀티에이전트 오케스트레이션을 통해 AI의 강점(빠른 발산, 병렬 처리, 반복 자동화)을 극대화하면서 약점(자기 확인 편향, 맥락 오염)을 구조적으로 억제한 점이 핵심 교훈이다.

향후 실제 학원 데이터를 확보하면 합성 데이터 자기순환 문제를 해소하여 모델 검증의 실용성을 높이고, 접근성 및 국제화 등 프론트엔드 품질도 개선할 예정이다.
