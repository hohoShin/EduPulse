# AI 활용 리포트 3 — oh-my-claudecode(OMC) 도구별 활용 전략

> **작성일:** 2026-04-08
> **사용 도구:** Claude Code (Claude Opus 4.6) + oh-my-claudecode v4.9.3
> **목적:** 멀티에이전트 오케스트레이션 도구(OMC)의 에이전트별 활용 전략, 데이터 흐름, 토큰 최적화 기법 정리

---

## 1. OMC 아키텍처 개요

oh-my-claudecode(OMC)는 Claude Code 위에서 동작하는 **멀티에이전트 오케스트레이션 레이어**이다. 단일 AI 모델을 역할별 전문 에이전트로 분할하여 복잡한 소프트웨어 엔지니어링 작업을 병렬/순차 처리한다.

```
                        ┌─────────────────┐
                        │   사용자 요청     │
                        └────────┬────────┘
                                 │
                        ┌────────▼────────┐
                        │  Claude Code    │
                        │  (Orchestrator) │
                        └────────┬────────┘
                                 │
              ┌──────────────────┼──────────────────┐
              │                  │                  │
     ┌────────▼──────┐  ┌───────▼───────┐  ┌───────▼───────┐
     │   Planner     │  │   Architect   │  │    Executor   │
     │   (계획)       │  │   (설계 검증)  │  │    (구현)      │
     └───────────────┘  └───────────────┘  └───────────────┘
              │                  │                  │
     ┌────────▼──────┐  ┌───────▼───────┐  ┌───────▼───────┐
     │    Critic     │  │   Verifier    │  │  Code-Reviewer│
     │   (품질 평가)  │  │   (검증)      │  │   (코드 리뷰)  │
     └───────────────┘  └───────────────┘  └───────────────┘
```

### 핵심 설계 원칙

1. **관심사 분리:** 계획/구현/검증을 다른 에이전트가 담당 — 자기 승인 방지
2. **모델 라우팅:** 작업 복잡도에 따라 Haiku(경량) / Sonnet(표준) / Opus(심층 분석) 자동 배정
3. **컨텍스트 격리:** 서브에이전트는 독립 컨텍스트에서 실행 — 메인 컨텍스트 오염 방지

## 2. 에이전트별 페르소나 및 활용 전략

### 2.1 Planner (계획 수립 에이전트)

| 항목 | 내용 |
|---|---|
| **모델** | Opus (깊은 분석 필요) |
| **페르소나** | 전략 컨설턴트. 요구사항을 분석하고 실행 가능한 단계별 계획으로 변환 |
| **입력** | 프로젝트 문서(README, CLAUDE.md), 사용자 제약 조건 |
| **출력** | RALPLAN-DR 요약 + Phase별 파일 목록/수락 기준 + ADR |

**활용 전략:**
- 계획서에 반드시 **수락 기준(Acceptance Criteria)**을 bash 커맨드로 포함시켜 검증 가능성 확보
- Open Questions를 명시적으로 분리하여 사용자 의사결정을 구조화
- ADR(Architecture Decision Record) 형식으로 결정 근거를 문서화하여 재현성 확보

**제약 사항:**
- 코드를 직접 작성하지 않음 (계획만 수립)
- 사용자 확인 없이 실행 단계로 넘어가지 않음

---

### 2.2 Architect (아키텍처 검증 에이전트)

| 항목 | 내용 |
|---|---|
| **모델** | Opus (READ-ONLY, 수정 불가) |
| **페르소나** | 시니어 아키텍트. 기술적 건전성을 검증하고 가장 강력한 반론을 제시 |
| **입력** | Planner의 계획서, 프로젝트 소스 코드 |
| **출력** | Steelman 반론 + Tradeoff 분석 + 구체적 권고 + APPROVE/ITERATE/REJECT |

**활용 전략:**
- **Steelman Antithesis 필수:** 계획에 대한 가장 강력한 반대 논거를 먼저 구성 → 약점을 사전에 드러냄
- **Tradeoff Tension:** 계획이 충분히 다루지 않은 실질적 긴장을 최소 1개 이상 식별
- **원칙 위반 탐지:** 계획서 자체의 원칙과 실행 내용 간 모순 검출

**제약 사항:**
- 코드를 수정할 수 없음 (READ-ONLY) — 의도적 설계로 검토의 객관성 보장
- Critic보다 반드시 먼저 실행 (Critic은 Architect 결과를 참조)

**본 프로젝트 적용 사례:**
- Droplet 2GB에서 3모델 동시 서빙 불가 문제를 계획 단계에서 발견
- 합성 데이터 자기순환 문제 지적 → 3모델 비교에서 단일 모델 집중으로 전략 전환

---

### 2.3 Critic (품질 평가 에이전트)

| 항목 | 내용 |
|---|---|
| **모델** | Opus (READ-ONLY, 수정 불가) |
| **페르소나** | QA 리드. 5가지 품질 기준으로 계획을 체계적으로 평가 |
| **입력** | Planner 계획서 + Architect 리뷰 결과 |
| **출력** | 기준별 Pass/Fail 판정 + 누락 항목 + Multi-Perspective 분석 + APPROVE/ITERATE/REJECT |

**활용 전략:**
- **5가지 품질 기준 강제 적용:**
  1. 원칙-옵션 일관성
  2. 대안 공정 평가
  3. 리스크 완화 명확성
  4. 검증 가능한 수락 기준
  5. 구체적 검증 절차
- **Multi-Perspective:** Executor(실행자), Stakeholder(이해관계자), Skeptic(회의론자) 3가지 관점에서 교차 검증
- Architect가 놓친 이슈를 독립적으로 탐지 (예: `pydantic-settings` 누락)

**제약 사항:**
- Architect 완료 후에만 실행 (순차 의존성)
- 동일 컨텍스트에서 자기 승인 불가 — Planner와 분리된 에이전트로 실행

---

### 2.4 Executor (구현 에이전트)

| 항목 | 내용 |
|---|---|
| **모델** | Sonnet (표준) / Opus (복잡한 구현) |
| **페르소나** | 시니어 개발자. 계획서를 코드로 변환. 파일 생성/수정/삭제 가능 |
| **입력** | 승인된 계획서 + 태스크 목록 |
| **출력** | 실제 코드 파일, 테스트, 설정 파일 |

**활용 전략:**
- 계획서의 수락 기준을 그대로 테스트로 변환
- 한 번에 하나의 Task에 집중 (Task 완료 → 다음 Task)
- `run_in_background`로 빌드/테스트를 백그라운드 실행하여 병렬성 확보

**제약 사항:**
- 계획서에 없는 범위를 임의로 확장하지 않음
- 보안 민감 파일(.env, credentials)은 직접 커밋하지 않음

---

### 2.5 Code-Reviewer (코드 리뷰 에이전트)

| 항목 | 내용 |
|---|---|
| **모델** | Sonnet / Opus |
| **페르소나** | 코드 리뷰어. 심각도 등급(Critical/Major/Minor)으로 피드백 |
| **입력** | 구현된 코드 diff |
| **출력** | 심각도별 피드백, SOLID 원칙 검증, 성능/보안 이슈 |

**활용 전략:**
- 작성과 리뷰를 반드시 분리 (Executor가 작성 → Code-Reviewer가 리뷰)
- 로직 결함 탐지에 집중 (스타일 이슈보다 우선)
- 보안 취약점(OWASP Top 10) 자동 스캔

---

### 2.6 Verifier (검증 에이전트)

| 항목 | 내용 |
|---|---|
| **모델** | Sonnet |
| **페르소나** | QA 엔지니어. 증거 기반 완료 확인 |
| **입력** | 수락 기준 목록 + 테스트 결과 |
| **출력** | 기준별 Pass/Fail + 증거(로그, 스크린샷) |

**활용 전략:**
- 각 Phase 완료 시 수락 기준을 하나씩 실행하여 Pass/Fail 판정
- "완료"를 주장하기 전에 반드시 Verifier를 거침
- 실패 시 Executor에게 수정 요청 → 재검증 루프

---

### 2.7 Explore (코드베이스 탐색 에이전트)

| 항목 | 내용 |
|---|---|
| **모델** | Haiku (빠른 탐색) / Sonnet |
| **페르소나** | 코드베이스 내비게이터. 파일/패턴 검색 전문 |
| **입력** | 검색 키워드, 파일 패턴, 질문 |
| **출력** | 관련 파일 목록, 코드 스니펫, 구조 설명 |

**활용 전략:**
- 단순 검색(특정 파일/함수)은 Glob/Grep 직접 사용으로 토큰 절약
- 광범위한 코드베이스 탐색(3회 이상 쿼리 필요)시에만 Explore 에이전트 호출
- `thoroughness` 레벨 조절: quick(기본) / medium / very thorough

---

### 2.8 Document-Specialist (문서 전문 에이전트)

| 항목 | 내용 |
|---|---|
| **모델** | Sonnet |
| **페르소나** | 외부 문서/레퍼런스 전문가. 공식 문서 조회 후 정확한 사용법 제공 |
| **입력** | 라이브러리/API 이름, 사용법 질문 |
| **출력** | 공식 문서 기반 사용 예제, 버전별 차이점 |

**활용 전략:**
- SDK/프레임워크 사용법이 불확실할 때 우선 호출 (추측 코딩 방지)
- 공식 문서 → 웹 검색 순서로 fallback

## 3. 워크플로우별 에이전트 구성

### 3.1 Ralplan (합의 기반 계획 수립)

본 프로젝트에서 실제 사용한 워크플로우이다.

```
사용자 요청
    │
    ▼
[Planner] ──── 초안 작성 (RALPLAN-DR + Phase + ADR)
    │
    ▼
[Architect] ── 아키텍처 리뷰 (Steelman 반론, Tradeoff)
    │
    ▼                          ※ 순차 실행 필수
[Critic] ───── 품질 평가 (5기준 Pass/Fail)
    │
    ├── APPROVE → 실행 착수
    │
    └── ITERATE → Planner 수정 → Architect → Critic (최대 5회)
```

**데이터 흐름:**
1. 사용자 제약 조건 → Planner (입력)
2. Planner 출력(계획서.md) → Architect (입력)
3. Architect 출력(리뷰 결과) → Critic (입력)
4. Critic 판정 → ITERATE일 경우 Architect+Critic 피드백 → Planner (루프)

**토큰 사용량 (본 프로젝트 실측):**

| 단계 | 토큰 | 비고 |
|---|---|---|
| Planner 초안 | ~47K | 1,057행 계획서 생성 |
| Architect 리뷰 | ~60K | 코드베이스 탐색 포함 |
| Critic 평가 | ~69K | 다중 파일 검증 |
| Planner 수정 | ~65K | 피드백 반영 재작성 |
| Architect 재검토 | ~50K | 이전 이슈 해소 확인 |
| Critic 재평가 | ~61K | 최종 품질 검증 |
| **합계** | **~352K** | 2회 반복 |

---

### 3.2 Ralph (자기참조 실행 루프)

```
[Executor] ── 구현
    │
    ▼
[Verifier] ── 검증
    │
    ├── Pass → 완료
    └── Fail → Executor 재시도 (루프)
```

**용도:** 명확한 구현 지시가 있을 때 (파일 경로, 함수명, 테스트 기준이 확정된 상태)

---

### 3.3 Team (병렬 팀 실행)

```
[Team Lead] ── TaskList 관리
    │
    ├── [Agent A] ── Task 1 (Backend)
    ├── [Agent B] ── Task 2 (Frontend)
    └── [Agent C] ── Task 3 (Tests)
```

**용도:** 독립적인 태스크가 3개 이상일 때 병렬 실행으로 시간 단축

---

### 3.4 Autopilot (완전 자율 실행)

```
사용자 요청 → [Plan] → [Execute] → [Verify] → 완료
```

**용도:** 명확한 요구사항 + 충분한 컨텍스트가 있을 때 한 번에 끝까지 실행

## 4. 모델 라우팅 전략

| 모델 | 토큰 비용 | 속도 | 용도 |
|---|---|---|---|
| **Haiku** | 최저 | 최빠름 | 단순 검색, 파일 탐색, 형식 변환, 짧은 코드 생성 |
| **Sonnet** | 중간 | 보통 | 표준 구현, 코드 리뷰, 테스트 작성, 문서화 |
| **Opus** | 최고 | 느림 | 아키텍처 설계, 복잡한 디버깅, 심층 분석, 계획 수립 |

**라우팅 규칙:**
- 계획/설계/리뷰: **항상 Opus** (품질이 비용보다 중요)
- 코드 구현: **Sonnet 기본**, 복잡한 로직은 Opus
- 탐색/검색: **Haiku 우선** (빠른 응답이 중요)
- Writer(문서): **Haiku** (비용 효율)

## 5. 토큰 최적화 전략

### 5.1 컨텍스트 격리

```
메인 컨텍스트 (사용자 대화)
    │
    ├── 서브에이전트 A (독립 컨텍스트) ── 결과만 반환
    ├── 서브에이전트 B (독립 컨텍스트) ── 결과만 반환
    └── 서브에이전트 C (독립 컨텍스트) ── 결과만 반환
```

- 서브에이전트는 **독립 컨텍스트**에서 실행되어 메인 컨텍스트를 오염시키지 않음
- 대용량 파일 탐색, 광범위한 코드 검색은 서브에이전트에 위임하여 메인 컨텍스트 토큰 절약
- 결과만 요약되어 메인 컨텍스트로 반환

### 5.2 도구 선택 최적화

| 상황 | 비효율적 | 효율적 |
|---|---|---|
| 특정 파일 읽기 | Agent(Explore) 호출 | **Read** 직접 사용 |
| 파일명 검색 | Bash `find` 실행 | **Glob** 패턴 매칭 |
| 코드 내 검색 | Bash `grep` 실행 | **Grep** 전용 도구 |
| 3회 이상 탐색 | Glob/Grep 반복 | **Explore 에이전트** 1회 호출 |

**원칙:** 전용 도구가 있으면 Bash 대신 사용. 3회 미만 검색은 직접 도구 사용, 3회 이상은 에이전트 위임.

### 5.3 병렬 실행

```python
# 나쁜 예: 순차 실행 (토큰 + 시간 낭비)
result_a = agent_a.run()  # 대기
result_b = agent_b.run()  # result_a 불필요한데 대기

# 좋은 예: 병렬 실행 (독립 작업은 동시 수행)
result_a, result_b = parallel(agent_a.run(), agent_b.run())
```

- 독립적인 도구 호출은 **단일 메시지에 여러 도구를 동시 호출**
- `run_in_background` 플래그로 빌드/테스트를 백그라운드 실행
- 의존성이 있는 호출만 순차 실행 (예: Architect → Critic)

### 5.4 ignorePaths 설정

```json
// .claude/settings.json
{
  "ignorePaths": ["docs/", "node_modules/", ".git/"]
}
```

- 불필요한 디렉토리를 탐색 대상에서 제외하여 토큰 낭비 방지
- 대용량 문서, 빌드 산출물, 버전 관리 파일 등

### 5.5 메모리 시스템 활용

```
~/.claude/projects/{project}/memory/MEMORY.md  ← 항상 로드됨 (200행 제한)
~/.claude/projects/{project}/memory/*.md       ← 주제별 상세 메모
```

- 반복되는 프로젝트 컨텍스트를 메모리에 저장 → 매 대화 시 재설명 불필요
- 검증된 패턴, 아키텍처 결정, 디버깅 해법을 저장하여 중복 탐색 방지
- 세션별 임시 컨텍스트는 저장하지 않음 (메모리 오염 방지)

## 6. 유지보수성 및 재현성 전략

### 6.1 CLAUDE.md 기반 프로젝트 컨텍스트

```
CLAUDE.md (프로젝트 루트)
├── 프로젝트 개요
├── 기술 스택
├── 디렉토리 구조
├── 코딩 컨벤션
├── 환경 전략
└── 공통 실수 (Common Pitfalls)
```

- **모든 에이전트가 자동으로 CLAUDE.md를 읽음** → 프로젝트 컨텍스트 일관성 보장
- 새로운 에이전트가 투입되어도 CLAUDE.md만으로 프로젝트 파악 가능
- 아키텍처 결정이 변경되면 CLAUDE.md를 업데이트하여 모든 에이전트에 전파

### 6.2 계획서의 재현성

| 요소 | 재현성 확보 방법 |
|---|---|
| 결정 근거 | ADR(Architecture Decision Record) 형식으로 문서화 |
| 대안 분석 | RALPLAN-DR로 구조화 (원칙, 결정 요인, 옵션 비교) |
| 수락 기준 | bash 커맨드로 작성 → 누구든 동일하게 검증 가능 |
| 리뷰 이력 | Architect/Critic 피드백 + 반영 내역을 문서로 보존 |
| 미결 사항 | Open Questions 파일로 분리 → 해결 상태 추적 |

### 6.3 에이전트 분리의 유지보수 효과

```
문제: Executor가 작성한 코드에 버그가 있음
기존: 같은 에이전트가 작성 + 리뷰 → 자기 맹점 재현
OMC: Executor 작성 → Code-Reviewer 독립 리뷰 → 맹점 탐지
```

- 작성과 검증을 분리함으로써 **확인 편향(confirmation bias)** 방지
- 각 에이전트의 역할이 명확하여 문제 발생 시 어느 단계에서 실패했는지 추적 가능

## 7. EduPulse 프로젝트 실제 적용 요약

| 단계 | 사용한 에이전트 | 모델 | 효과 |
|---|---|---|---|
| 계획 수립 | Planner | Opus | 4 Phase 구조화, 일별 태스크 분해 |
| 아키텍처 검증 | Architect | Opus | 메모리 초과 사전 발견, 모델 전략 전환 |
| 품질 평가 | Critic | Opus | pydantic-settings 누락 등 블로커 사전 발견 |
| 요구사항 확인 | AskUserQuestion | - | 7개 미결 질문 → 제약 조건 확정 |
| docs 제외 설정 | ignorePaths | - | 불필요한 파일 탐색 토큰 절약 |
| 계획 수정 | Planner (재호출) | Opus | 피드백 14/15건 반영 |
| 최종 검증 | Architect + Critic | Opus | 2회 반복으로 합의 달성 |

**총 에이전트 호출:** 6회 (Planner 2 + Architect 2 + Critic 2)
**합의 반복 횟수:** 2회 (최대 5회 허용 중)
**발견된 이슈:** 15건 (Critical 5 + Major 5 + Minor 5)
**사전 방지된 블로커:** 2건 (pydantic-settings, main.py 마이그레이션)
