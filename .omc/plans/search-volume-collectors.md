**Status:** REVISED (RALPLAN re-review incorporating Architect + Critic feedback)
**Complexity:** MEDIUM

## RALPLAN-DR Summary

### Principles (5)

1. **Naver-primary, Google-supplementary**: Naver DataLab is the authoritative data source. Google Trends is cached locally for future research only and NEVER appears in pipeline output.
2. **Match codebase style**: Module-level functions (no ABC/class hierarchy). Follow patterns in `external_generator.py`, `merger.py`, `transformer.py`.
3. **Atomic and safe**: All file writes use atomic temp-file-then-rename. Quota tracking is persistent. Failures never corrupt existing data.
4. **Transparent fallback**: Naver API fail -> cached Naver data -> skip (log warning). Google never substitutes for Naver in output.
5. **Zero downstream changes**: Output schema matches `external_generator.py` exactly (`date, field, search_volume, ds, y`). `merger.py` requires no modifications.

### Decision Drivers (top 3)

1. **Pipeline compatibility**: `merger.py:37` selects `[field, date, search_volume]` from `search_trends.csv`. `merger.py:74` reads from `raw_external_dir`. Output must be drop-in compatible.
2. **Naver API quota protection**: Naver DataLab has a daily call limit. Quota must persist across runs, reset at 00:00 KST, and be tracked atomically.
3. **Testability without live APIs**: All API calls must be mockable. Tests must never hit real endpoints.

### Viable Options

**Option A (CHOSEN): Layered module-level functions in `edupulse/collection/api/`**
- Pros: Matches codebase style, clear separation (naver/google/orchestrator), easy to test individual functions
- Cons: No shared interface enforcement (mitigated by consistent function signatures and tests)

**Option B (REJECTED): Single monolithic collector module**
- Pros: Fewer files, simpler imports
- Cons: Violates separation of concerns, harder to test Naver vs Google independently, harder to maintain quota logic
- **Invalidation rationale**: Mixing Naver API logic, Google scraping, caching, quota tracking, and orchestration in one file would exceed 400 lines and make unit testing impractical. The codebase already separates concerns (see `preprocessing/cleaner.py`, `transformer.py`, `merger.py` as separate modules).

### ADR

- **Decision**: Implement as 5 module-level-function files under `edupulse/collection/api/`
- **Drivers**: Pipeline compatibility, quota safety, testability
- **Alternatives considered**: Monolithic module (rejected for testability), ABC-based classes (rejected for codebase style mismatch)
- **Why chosen**: Best balance of codebase consistency, testability, and maintainability
- **Consequences**: 5 new files + 1 test file + 2 modified files. No breaking changes to existing pipeline.
- **Follow-ups**: Job posting crawler (`edupulse/collection/crawlers/job_posting.py`) can follow the same pattern later.

## Context

The EduPulse pipeline currently uses synthetic search trend data generated by `edupulse/data/generators/external_generator.py` and written to `edupulse/data/raw/external/search_trends.csv` by `run_all.py`. The downstream `merger.py` reads this CSV and joins on `[field, date, search_volume]`.

This plan replaces the synthetic data source with real Naver DataLab API collection (primary) and adds Google Trends caching (supplementary, not used in pipeline output). The real collector writes to the SAME path (`edupulse/data/raw/external/search_trends.csv`), making it a drop-in replacement. The synthetic generator remains available for testing/development.

## Guardrails

### Must Have
- Output schema: `date, field, search_volume, ds, y` (exactly matches `external_generator.py`)
- Output path: `edupulse/data/raw/external/search_trends.csv` (exactly matches `merger.py:74`)
- Naver = sole data source for `search_volume` in output
- Atomic writes for CSV and quota file (write to `.tmp`, then `os.replace()`)
- Exponential backoff retry: 1s / 2s / 4s, cap 60s, max 3 attempts
- Quota persistence across runs, reset at 00:00 KST
- Python `logging` module throughout (no `print()`)
- All API calls mockable in tests
- Cache staleness warning when cached data is older than 4 weeks

- No ABC or class-based hierarchy
- No `source` column in output CSV
- No Google Trends data in `search_trends.csv` output
- No mixing of Naver + Google data in any output row
- No hardcoded API keys (use `edupulse/config.py` Settings)
- No changes to `merger.py`
- No changes to `external_generator.py` or `run_all.py`

## Task Flow (6 Steps)

### Step 1: Keyword Config + Quota Tracker

**Files:** `edupulse/collection/__init__.py`, `edupulse/collection/api/__init__.py`, `edupulse/collection/api/keywords.py`, `edupulse/collection/api/quota.py`

**keywords.py** contains:
- `FIELD_KEYWORDS` dict (Korean, for Naver)
- `FIELD_KEYWORDS_EN` dict (English, for Google)
- `AGGREGATION_METHOD = "sum"` constant

**quota.py** contains:
- `QUOTA_FILE = "edupulse/data/raw/external/.naver_quota.json"` path constant
- `read_quota() -> dict` - Reads quota JSON `{"date": "YYYY-MM-DD", "calls_today": int}`. Returns `{"date": today_kst, "calls_today": 0}` if file missing or date differs from today (KST).
- `increment_quota() -> int` - Atomically increments and returns new count. Uses KST (UTC+9) for date boundary. Atomic write: write `.tmp` then `os.replace()`.
- `check_quota(max_daily: int) -> bool` - Returns True if calls_today < max_daily.
- KST calculation: `datetime.now(timezone(timedelta(hours=9)))` (no pytz dependency)

**Acceptance criteria:**
- `read_quota()` returns `{"date": "2026-04-08", "calls_today": 0}` when file is missing
- `read_quota()` resets to 0 when stored date differs from today (KST)
- `increment_quota()` writes atomically (no partial writes on crash)
- `check_quota(1000)` returns False when calls_today >= 1000

### Step 2: Naver DataLab Client

**File:** `edupulse/collection/api/naver_datalab.py`

**Functions:**

```python
import logging
import httpx

logger = logging.getLogger(__name__)

def fetch_naver_trends(
    keywords: list[str],
    start_date: str,      # "YYYY-MM-DD"
    end_date: str,         # "YYYY-MM-DD"
    client_id: str,
    client_secret: str,
    time_unit: str = "week",
) -> list[dict]:
    """Call Naver DataLab Search Trend API.

    Returns list of {"period": "YYYY-MM-DD", "ratio": float} dicts per keyword.
    Raises NaverAPIError on non-200 response after retries.
    """

def fetch_field_trends(
    field: str,
    start_date: str,
    end_date: str,
    client_id: str,
    client_secret: str,
) -> pd.DataFrame:
    """Fetch and aggregate trends for one field.

    Calls fetch_naver_trends() for all keywords in FIELD_KEYWORDS[field].
    Aggregates by summing keyword volumes per week.
    Returns DataFrame with columns: date, field, search_volume, ds, y
    """

def fetch_all_fields(
    start_date: str,
    end_date: str,
    client_id: str,
    client_secret: str,
    max_daily_calls: int = 1000,
) -> pd.DataFrame | None:
    """Fetch trends for all 4 fields with quota enforcement.

    Checks quota before each API call. Stops and returns partial results
    if quota exhausted. Returns None if no data collected.
    Caches raw JSON responses per field.
    """
```

**Key behaviors:**
- Uses `httpx` (already in requirements.txt) for HTTP requests
- POST to `https://openapi.naver.com/v1/datalab/search` with headers `X-Naver-Client-Id`, `X-Naver-Client-Secret`
- Request body: `{"startDate": "YYYY-MM-DD", "endDate": "YYYY-MM-DD", "timeUnit": "week", "keywordGroups": [{"groupName": field, "keywords": [...]}]}`
- Retry with exponential backoff: delays = [1, 2, 4] seconds, max 3 attempts, cap 60s
- On HTTP 429 or 5xx: retry. On 4xx (except 429): raise immediately.
- Logs each API call at INFO level, errors at ERROR level
- Calls `increment_quota()` BEFORE each API request
- **Cache**: saves raw JSON response to `edupulse/data/raw/external/cache/naver_{field}_{start}_{end}.json`

**Date range strategy:**
- Default: trailing 52 weeks from today (1 year of weekly data)
- Caller can override with explicit start_date / end_date
- Naver DataLab supports max 5 years of history

**Acceptance criteria:**
- `fetch_naver_trends()` sends correct POST body to Naver API
- Retries on 429/5xx with exponential backoff (1s, 2s, 4s)
- Raises `NaverAPIError` after 3 failed attempts
- `fetch_field_trends()` returns DataFrame with columns `[date, field, search_volume, ds, y]`
- `search_volume` = sum of keyword ratios for the field
- `ds` = `date`, `y` = `search_volume` (Prophet compatibility)
- Quota is incremented before each request
- Raw response cached as JSON file

### Step 4: Orchestrator

**File:** `edupulse/collection/api/collect_search_trends.py`

**Functions:**

```python
import logging

logger = logging.getLogger(__name__)

def collect_search_trends(
    output_path: str = "edupulse/data/raw/external/search_trends.csv",
    cache_dir: str = "edupulse/data/raw/external/cache",
    start_date: str | None = None,
    end_date: str | None = None,
    max_daily_calls: int = 1000,
    also_cache_google: bool = False,
) -> str | None:
    """Main entry point: collect Naver search trends and write pipeline-compatible CSV.

    Fallback chain:
    1. Call Naver DataLab API -> success -> write to output_path
    2. Naver fails -> load most recent cached Naver JSON -> write to output_path
    3. No cache available -> skip (log warning, return None)

    Google Trends is NEVER used in output. If also_cache_google=True,
    Google data is fetched and cached separately for research.

    Returns output_path on success, None on skip.
    """

def normalize_to_weekly(df: pd.DataFrame, date_col: str = "date") -> pd.DataFrame:
    """Normalize daily or irregular dates to weekly (Monday start).

    Handles both daily and weekly input granularity.
    Groups by (field, week) and sums search_volume.
    """

def _load_cached_naver(
    cache_dir: str,
    field: str,
) -> pd.DataFrame | None:
    """Load most recent cached Naver JSON for a field. Returns None if no cache.

    Logs WARNING if cache is older than 4 weeks.
    """
```

**Fallback contract (explicit):**
```
Naver API call succeeds          -> use fresh data
Naver API call fails (all retry) -> load cached Naver JSON
  cached Naver exists            -> use cached data (log WARNING with cache age)
  cached Naver > 4 weeks old     -> use it anyway but log STALENESS WARNING
  no cached Naver exists         -> skip field (log WARNING)
Google Trends                    -> NEVER used in output, only cached separately
```

**Output format** (written atomically to `search_trends.csv`):
```csv
date,field,search_volume,ds,y
2026-01-05,coding,245,2026-01-05,245
2026-01-05,security,180,2026-01-05,180
...
```
- `date`: weekly Monday start date, YYYY-MM-DD
- `field`: one of `coding`, `security`, `game`, `art`
- `search_volume`: integer, sum of Naver keyword ratios for the field
- `ds`: same as `date` (Prophet compatibility)
- `y`: same as `search_volume` (Prophet compatibility)

**Invocation:**
```bash
# CLI invocation
python -m edupulse.collection.api.collect_search_trends

# Programmatic
from edupulse.collection.api.collect_search_trends import collect_search_trends
result = collect_search_trends()

# With Google caching
result = collect_search_trends(also_cache_google=True)
```

The `if __name__ == "__main__"` block calls `collect_search_trends()` with defaults from `edupulse.config.settings`.

**Acceptance criteria:**
- Writes to `edupulse/data/raw/external/search_trends.csv` (same path as `run_all.py`)
- Output has exactly columns: `date, field, search_volume, ds, y`
- `merger.py` can read the output without any changes
- Fallback chain works: API -> cache -> skip
- Atomic write (`.tmp` then `os.replace()`)
- Google data is never in output CSV even when `also_cache_google=True`
- `normalize_to_weekly()` handles both daily and weekly input
